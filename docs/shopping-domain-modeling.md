# 쇼핑몰 도메인 모델링

> 문서 관련 안내사항

처음 접하시는 분들의 이해를 돕기 위해 문서의 역할을 두 가지로 분리했습니다.
- 앞부분의 '용어 사전'은 기획자, 디자이너, 개발자 등 모든 직군이 동일한 이해를 바탕으로 **소통**하기 위해 존재합니다.
- 뒷부분의 '모델링 하기'는 개발자가 실제 객체와 시스템을 **구현**하기 위한 명세서 역할을 합니다.

두 파트에 유사한 개념이 등장하더라도, 각각 소통과 구현이라는 다른 목적을 위해 작성되었다는 점을 참고해 주시기 바랍니다.

## 용어 사전 만들기

> 용어 사전은 프로젝트(또는 서비스)에 참여하는 도메인 전문가, 개발자, 기획자, 디자이너 등 다양한 직군 간의 의사소통을 명확하게 하기 위한 도구입니다.
> 이 사전은 하위 도메인과 바운디드 컨텍스트를 기준으로 도메인 내 개념을 일관되게 설명합니다.

하위 도메인은 **자연스럽게 나뉘는 주제** 영역입니다.

- 비즈니스 전체 흐름 속에서 각 영역이 무엇을 책임지는지에 따라 구분됩니다.

바운디드 컨텍스트는 시스템이 책임지는 논리적 경계입니다. (=개발자가 **인위적으로 경계를 나누는 것**)

- 개발자가 일하기 편하게 쪼갠 구현 단위이며, 내부에는 의미 있는 도메인 객체들이 존재하고 상호작용합니다.

### 하위 도메인

- `Member`: 사용자 가입 및 인증
- `Product`: 상품 정보 및 검증 관리
- `Wish`: 사용자의 관심 상품 관리

### 바운디드 컨텍스트

| 바운디드 컨텍스트 | 설명                                                                 |
|-----------|--------------------------------------------------------------------|
| Member    | 회원 가입 및 로그인 요청을 처리하며, 인증된 사용자를 식별하기 위한 토큰을 발급하고 검증하는 책임을 갖는다.      |
| Product   | 상품의 생명주기(조회, 추가, 수정, 삭제)를 관리하며, 도메인 규칙 및 외부 API를 통한 비속어 필터링을 담당한다. |
| Wish      | 특정 회원이 관심 있어 하는 상품의 목록(위시 리스트)을 매핑하고 관리한다.                         |

#### Member (회원 및 인증)

| **한글명** | **영문명**     | **설명**                                   |
|---------|-------------|------------------------------------------|
| 회원 ID   | memberId    | 내부 시스템에서 식별하는 회원의 고유 식별자                 |
| 이메일     | email       | 로그인 시 사용하는 식별자                           |
| 비밀번호    | password    | 회원 인증을 위한 비밀번호 (보안을 위해 암호화 저장)           |
| 인증 토큰   | accessToken | 로그인 성공 시 발급되는 문자열로, API 접근 권한을 증명하는 데 사용 |

#### Product (상품)

| **한글명** | **영문명**   | **설명**                       |
|---------|-----------|------------------------------|
| 상품 ID   | productId | 상품의 고유 식별자                   |
| 상품 이름   | name      | 상품의 명칭 (최대 15자, 특정 특수문자만 허용) |
| 상품 가격   | price     | 상품의 판매 가격                    |
| 이미지 URL | imageUrl  | 이미지가 호스팅된 외부 주소 문자열          |

#### Wish (위시 리스트)

| **한글명** | **영문명**   | **설명**                |
|---------|-----------|-----------------------|
| 위시 ID   | wishId    | 위시 리스트 항목의 고유 식별자     |
| 회원 ID   | memberId  | 해당 위시 리스트를 소유한 회원 식별자 |
| 상품 ID   | productId | 위시 리스트에 담긴 대상 상품 식별자  |

## 모델링 하기

### Member (회원)

#### 속성(상태)

- 회원은 이메일과 비밀번호를 가진다.

#### 행위(기능)

- 이메일과 비밀번호로 회원가입을 할 수 있다.
- 이메일과 비밀번호를 검증하여 로그인을 수행하고 인증 토큰을 발급한다.

#### 정책(제약사항)

- 로그인 시 가입된 이메일과 비밀번호가 일치해야 한다.
- 생성된 인증 토큰은 이후 보호된 API(위시 리스트 등) 접근 시 검증 수단으로 사용된다.

### Product (상품)

#### 속성(상태)

- 상품은 이름, 가격, 이미지 URL을 가진다.

#### 행위(기능)

- 상품을 생성, 조회(특정 상품/모든 상품), 수정, 삭제할 수 있다.

#### 정책(제약사항)

- 데이터 형식 제한:
    - 상품 이름은 공백을 포함하여 최대 15자까지만 입력할 수 있다.
    - 상품 이름에 허용된 특수문자(`(`, `)`, `[`, `]`, `+`, `-`, `&`, `/`, `_`) 외에는 사용할 수 없다.
- 비속어 차단:
    - 상품 추가/수정 시 `PurgoMalum API`를 호출하여 비속어가 포함된 경우 저장을 차단하고 오류 응답을 반환해야 한다.
- 이미지 처리:
    - 파일 업로드는 허용하지 않으며, 반드시 URL 문자열 형태로만 입력받는다.

### Wish (위시 리스트)

#### 속성(상태)

- 특정 회원의 ID와 등록된 상품의 ID로 구성된다.

#### 행위(기능)

- 위시 리스트에 상품을 추가할 수 있다.
- 위시 리스트에 담긴 상품 목록을 조회할 수 있다.
- 위시 리스트에서 특정 상품을 삭제할 수 있다.

#### 정책(제약사항)

- 인증 필수
    - 위시 리스트의 모든 기능(조회/추가/삭제)은 인증 토큰을 가진 회원만 사용할 수 있다.
- 소유권 검증:
    - 위시 리스트 조회 및 삭제는 본인이 추가한 내역에 대해서만 접근이 가능해야 한다.
- 존재하는 상품(Product)에 대해서만 위시 리스트 추가가 가능하다.
