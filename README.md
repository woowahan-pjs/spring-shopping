# spring-shopping

## 기능 요구사항
HTTP 요청과 응답은 JSON 형식으로 주고받으며, 스프링 프레임워크를 사용하여 웹 애플리케이션으로 구현한다.

### 상품
상품을 조회, 추가, 수정, 삭제하는 기능을 구현한다.
- [] 상품에는 이름, 가격, 이미지 URL이 있다.
  - [] 상품 이름은 공백을 포함하여 최대 15자까지 입력할 수 있다.
  - [] 상품 이름에는 아래 특수 문자만 사용할 수 잇다.
    - [] 가능 : (), [], +, -, &, /, _
    - [] 그외 특수문자는 사용할 수 없다.
  - [] 상품의 이름에는 비속어가 포함될 수 없다.
      - [] PurgoMalum API를 호출하여 비속어 포함 여부를 검사한다.
  - [] 상품 가격은 0원부터 9억 9999만 9999원까지만 가능하다.
    - [] 가격은 마이너스가 될 수 없다.
  - [] 상품 이미지 URL
- [] 상품 이미지는 파일업로드가 아닌 URL 입력 방식으로 한다.

### 회원 로그인
회원 가입과 로그인을 통해 사용자 인증 기능을 구현한다.
- [] 회원은 이메일과 비밀번호로 가입할 수 있다.
  - [] 올바른 이메일 형식인지 확인한다.
  - [] 중복된 이메일인지 확인한다.
  - [] 이메일과 비밀번호는 비어있지 않아야한다.
  - [] 비밀 번호는 최소 8자 최대 20자로 제한을 둔다.
- [] 로그인 시 이메일과 비밀번호를 검증하고, 일치하면 인증 토큰을 발급한다.
- [] 토큰 생성 방식은 자유롭게 선택한다.

### 위시 리스트
인증된 사용자를 대상으로 위시 리스트 기능을 구현한다
- [] 위시 리스트에 등록된 상품 목록을 조회할 수 있다.
- [] 위시 리스트에 상품을 추가할 수 있다
  - [] 상품 리스트에서 추가할 수 있다.
  - [] 상품 상세에서 추가할 수 있다.
  - [] 존재하는 상품인지 확인해야한다.
- [] 위시 리스트에서 상품을 삭제 할 수 있다.
  - [] 로그인 한 사용자의 아이템이 맞는지 확인한다.


## 구현 전략

### 1. 아키텍처 및 설계 원칙
- 계층형 아키텍처 : Controller, Service, Repository로 역할을 분리하여 유지보수성을 높인다.
- 인터페이스 기반 설계 : 비즈니스 로직과 데이터 접근 기술을 분리하여 유연한 변경이 가능하도록 설계
- TDD : 테스트코드를 먼저 작성하고 기능을 구현하여 안정적으로 구축

### 2. 기술 스택
- Spring boot : 생산성 향상 및 의존성 관리 용이
- Java 21 : 최신 언어 기능 활용 및 간결한 코드 작성 
- JPA : 객체 지향적인 데이터 모델링과 생산성 향상
- H2 Database : 개발 및 테스트 환경에서의 편의성
- Spring Security & JWT : 확장성 있는 인증/인가 시스템 구축

### 3. 단계별 구현 계획
- 1. 핵심 도메인 모델링 및 InMemory Repository 구현
  - 상품, 회원, 위시리스트 순으로 도메인 객체 설계
- 2. 비즈니스 로직 구현 및 테스트
  - Service 계층에서 비즈니스 규칙 검증 및 로직 수행.
  - JUnit 5를 활용한 단위 테스트 작성
- 3. 웹 계층 구현 
  - RESTful API 설계 및 Controller 구현
  - MockMvc를 활용한 통합 테스트 작성
- 4. 데이터 영속성 전환
  - Repository 인터페이스를 유지하며 구현체를 JPA로 교체.
  - 트랜잭션 관리 적용.
- 5. 보안 강화 
  - 인증/인가 로직을 Spring Security Filter Chain으로 이관
  - JWT 기반의 Stateless 인증 구현
